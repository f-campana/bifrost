01 Overview

A hybrid human-orchestrated agent operating system built from a clean slate. Two shared infrastructure layers — Sanctum and Bifrost — and any number of product consumers.

Sanctum
The compiled knowledge wiki. Owns everything the system knows — sources, synthesis, memory, indexes, project context, and the design identity library. Maintained by agents, read by humans and agents. Knowledge compounds over time.
Bifrost
The schema and operating system that governs agent behaviour. Owns skills, contracts, routing rules, verification logic, and prompt assembly rules. Projects consume both Sanctum and Bifrost.
Runestone
The browsable web interface to Sanctum's design identity library. Renders each Rune as a live experience and exposes the routing guide. Built using the design pipeline itself — first dogfood project.

The orchestrator is a human. Automation is earned, not assumed. Intelligence lives at the top, not in a routing layer. No AI orchestrator. No agents spawning agents without human dispatch.

The Karpathy model

Raw sources Sanctum/40 Sources/ immutable — LLM reads, never writes Compiled wiki Sanctum/10 Knowledge/ etc. LLM-maintained synthesis, compounds over time Schema Bifrost/ operating instructions and skills Key property: the wiki is a persistent compounding artifact. Good answers become wiki pages. Nothing useful disappears into chat history.
02 Core Principles
Human orchestrator
All routing decisions, task decompositions, and promote gate judgments belong to the human. Automation is for execution, not orchestration.
Sources are immutable
Raw source material in 40 Sources/ is never modified by agents. Provenance must be traceable. Errors stay in sources; corrections live in the compiled zone.
The wiki compounds
Every useful synthesis gets filed back into the wiki. Nothing valuable disappears into chat history. The system gets smarter with every interaction.
Bifrost ≠ knowledge
Bifrost defines how the wiki is maintained. Sanctum contains what the wiki knows. These are different concerns and must never blur.
Pull before push
Every automated trigger starts as manual dispatch. A role earns automated triggering only after ≥2 weeks of predictable manual use.
Dogfood the build
Each phase uses what was built in the previous phase. The design pipeline builds the Runestone. The wiki/ingest skill files the notes from its own creation.
Promote gate
Nothing moves from agent output to durable wiki without human review. Background agents write to 00 Inbox/ only.
Domain deepening
Each domain starts with a lightweight skill and earns a full contract through real use. When failures become specific, recurring, and consequential — invest in the deep process: research → model → validate → spec.

06 The Domain-Deepening Pattern

The system improves domain by domain. Each domain starts lightweight and earns depth through real use. This is the mechanism by which the system compounds in quality over time.

1 — Research
Find who has already solved this domain rigorously. Study their approach, vocabulary, and failure modes before writing anything. External research compresses months of trial and error into a structured starting point.
2 — Model
Identify the conceptual vocabulary for this domain. Name things precisely. Names must fit the broader system register and be yours — not borrowed from external systems.
3 — Validate
Run real work through lightweight specs. Observe what breaks specifically. The spec is written from evidence, not theory. Validation means real runs with observable, specific failures.
4 — Spec
Write the contract after validation. Not before. The design pipeline contract was written after four iterations of the builder and after understanding Ropau's system, Emil Kowalski's approach, and the Google spec format.

What qualifies a domain

Agent failures must be specific, recurring, and consequential. Generic dissatisfaction does not qualify. Specific observable patterns do.

07 Design Pipeline

Governs how aesthetic intent travels from a client brief into deployed UI. First fully-specced domain. Full specification: Bifrost/contracts/design-pipeline-contract.md.

Vocabulary

Rune
A named, versioned design identity. Encodes visual DNA, motion system, routing governance, and annotated radar. Lives in Sanctum/10 Knowledge/Design/runes/. Reusable across projects sharing the same archetype profile.
Runestone
The browsable web interface rendering the Rune library as a live experience: type specimens, colour swatches, motion previews, radar visualisation, selection guide. Phase 1 deliverable. Built by the design pipeline — first dogfood project.
Design Contract
Project-specific, agent-executable file instanced from a Rune. Lives at project root as DESIGN_[PROJECT].md. Google design.md spec compliant in YAML frontmatter. Self-standing — design/builder requires no other input.

08 Agent Taxonomy
Tier 1 — Actions
One domain. One verb. ≤10 files. Producers generate output. Validators report findings without modifying files. Validators never fix — they report.
Tier 2 — Workflows
Multiple actions chained within a domain. Routine (background): react-feature, api-endpoint, wiki-maintain, design-run. Complex (foreground): ddd-bounded-context, bdd-cycle.
Tier 3 — Projects
Multiple workflows across domains. Always foreground. Always human-initiated. full-stack-feature, wiki-build, runestone-build.
Tier 4 — Strategy
You. Full context. All routing decisions. The only tier that operates without a SKILL.md.

12 AGENTS.md Contract

Every repository that agents operate in has an AGENTS.md at the root. Required sections: Identity · Authorised actions · Quality gate · Skill loading · Context boot · Feedback channel.

The context boot principle: Agents do not arrive with context. AGENTS.md defines what to read first, in order. A well-written context boot means the agent begins already knowing the project architecture, quality standards, relevant skills, and current task context.

13 SKILL.md Contract

Every skill in Bifrost is a directory containing a single SKILL.md. Required frontmatter: name, version, type (producer/validator/workflow), domain, tier, triggers, reads_from, writes_to, model_preference.

Required body sections: When to use · What it does · What it does NOT do · Inputs · Output · Failure modes.

14 GitHub Skills

Read skills (Phase 1): read-file · read-diff · read-history · read-pr. No side effects.

Write skills (Phase 4): create-branch · commit-file · open-pr · comment-pr. All agent PRs open as draft. Loop-back must pass before any write skill runs.

15 Quality Gate

Bifrost/scripts/quality-gate.sh has two modes. --llm-only (fast, <30s): LLM lint surface + TypeScript type check, structured JSON output. --full: format + lint + typecheck + tests + build + changeset. Runs on pre-push and CI.

The LLM lint surface (eslint.llm.config.mjs) promotes agent-relevant warnings to errors: missing return types, any assertions, unused variables, type-unsafe patterns.

16 Loop-back Gate
Agent runs → quality-gate.sh --llm-only Exit 0: output moves to Inbox for human review Exit 1: structured feedback JSON → agent retry (max 3×) After 3 failures: escalate to human with failure JSON

Bifrost/scripts/loop-back.sh — ~50 lines. Built in Phase 2. Captures exit code and JSON from quality gate, formats feedback object, re-invokes agent with feedback prepended, tracks retry count.

17 Promote Gate + Self-Improvement

Nothing moves from 00 Inbox/ to durable wiki without human review. Agent writes to Inbox → human opens, reads, approves/rejects/edits → approved output moves to 10 Knowledge/ or 30 Memory/.

Self-improvement loop: run output → harvester/linter → harvest report in Inbox → human review → approved findings → distill → proposed Rune/wiki/skill update in Inbox → human commits. The system never commits improvements to itself.

18 The Core Operating Loop

Wiki: ingest (source → pages + index + log) · query (question → answer + page) · lint (health check → Inbox report).

Design: curator (brief + rune → Design Contract) · builder (contract → HTML + run report) · validator (HTML + contract → validation report) · harvester (run reports → harvest report).

Heartbeat (Phase 5+): Cron on Tachyon. Every 6h: wiki/lint. Every 24h: docs/drift + index/rebuild. Weekly: stale-scanner. Produces digest to Inbox + Slack notification.
